# Storage and Migration Timeline

## Schema Ownership

- `public` schema (owned by `leetcode-service`):
  - `problems`
  - `tags`
  - `problem_tags`
  - `test_cases`
- `submissions` schema (owned by `submissions` service):
  - `submissions.submissions`
  - `submissions.outbox`

Worker reads/writes both schema domains directly for evaluation path.

## Migration Timeline

### leetcode-service Flyway migrations

- `V1__create_tables.sql`
  - creates `problems`, `tags`, `problem_tags`
  - adds tag join index
- `V2__add_slug.sql`
  - adds `slug` to `problems`
  - backfills slug from URL
  - adds unique slug index
- `V3__create_test_cases_table.sql`
  - creates `test_cases`
  - adds `problem_id` index
- `V4__delete_problems_without_test_cases.sql`
  - removes problems with no available test cases

### submissions Flyway migrations

- `V1__create_schema.sql`
  - creates dedicated `submissions` schema
- `V2__create_submissions_table.sql`
  - creates `submissions.submissions`
  - tracks submission lifecycle state + JSONB result
  - indexes by `user_id` and `problem_id`
- `V3__create_outbox_table.sql`
  - creates transactional outbox table for Debezium CDC

## Key Table Contracts

### `submissions.submissions`

- Primary key: `id` (UUID, generated by DB)
- Status lifecycle begins at `PENDING`
- Result payload stored as JSONB once worker completes

### `submissions.outbox`

- Carries events to publish (currently `eval`)
- Critical columns:
  - `aggregate_id` (submission ID)
  - `event_type` (routing key semantics)
  - `payload` (EvalJob JSON)

### `test_cases`

- Foreign key: `problem_id -> problems.id`
- Ordered by `id` for stable worker execution order

## Data Access Pattern Summary

- `leetcode-service`: JDBC repository queries over public schema.
- `submissions`: JDBC repositories over submissions schema in service-owned transactions.
- `worker`: direct JDBC access to read test cases and persist submission results.
